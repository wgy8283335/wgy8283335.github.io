sharding-sphere-1.5.4.1源码分析-3

Posted on 2019-04-02 | Edited on 2019-04-04

在依赖了shardingjdbc后，通过下面的三个步骤来执行sql。
String sql = "SELECT i.* FROM t_order o JOIN t_order_item i ON o.order_id=i.order_id WHERE o.user_id=? AND o.order_id=?";
PreparedStatement preparedStatement = conn.prepareStatement(sql);
preparedStatement.executeQuery();
其中preparedStatement.executeQuery()即是调用ShardingPreparedStatement的executeQuery()方法。这里面的执行过程就是SQL查询的两个核心：路由和结果合并。
1.通过调用route()来解析SQL如何路由执行。
1.1里面调用了routingEngine.route(getParameters());内部调用PreparedStatementRoutingEngine中的route()方法，route()方法调用sqlRouter.route(logicSQL, parameters, sqlStatement)
1.1.1SQLRouter接口有两个实现类：DatabaseHintSQLRouter和ParsingSQLRouter，由于这里没有用hint语法强制执行某个库，所以调用ParsingSQLRouter中的route()方法。如果sql中只有一个表名(参与分库分表的表，数量为1时)，或者多个表名之间是绑定表关系（参与分库分表的表之间是绑定关系），或者所有表都在默认数据源指定的数据库中（即不参与分库分表的表），那么用SimpleRoutingEngine作为路由判断引擎；否则使用ComplexRoutingEngine
1.1.1.1SimpleRoutingEngine。根据逻辑表的名字取到相关的tablerule（数据源路由策略和表路由策略），然后根据tablerule得到查找得到相应的数据源，在数据源内再应用tablerule得到相应的物理表。将得到的路由数据源和表信息封装到RoutingResult中，RoutingResult中有个TableUnits类型属性，TableUnits类中有个List tableUnits属性，TableUnit包含三个属性：dataSourceName–数据源名称，logicTableName–逻辑表名称，actualTableName–实际表名称，例如：TableUnit:{dataSourceName:ds_jdbc_1, logicTableName:t_order, actualTableName: t_order_1}
1.1.1.2ComplexRoutingEngine
复杂路由引擎的核心逻辑就是拆分成多个简单路由，然后求笛卡尔积。
route函数中存放了核心逻辑。首先便利逻辑表集合，如果遍历的表配置了分库分表规则并且相关的绑定表没有被处理过，则对这个逻辑表构造简单路由规则，得到RoutingResult（含有三个属性：dataSourceName、logicTableName、actualTableName）。处理完当前表以后，寻找绑定表并将其加入绑定表的集合中（用于前面的判断），此处不产生新的RoutingResult，因为绑定表都是使用相同的路由信息。
确定一个SQL具体查询的物理表和数据源。比如一个数据库中查usertable_0,usertable_1;ordertable_0,ordertable_1。这两个表不是绑定的情况下，则需要求笛卡尔积。查usertable_0，ordertable_0；usertable_1，ordertable_0；usertable_0，ordertable_1；usertable_1，ordertable_1。

2.根据路由信息执行SQL。
List<ResultSet> resultSets = new PreparedStatementExecutor( getConnection().getShardingContext().getExecutorEngine(), routeResult.getSqlStatement().getType(), preparedStatementUnits, getParameters()).executeQuery();
在创建了PreparedStatementExecutor后执行executeQuery()，内部调用executorEngine.executePreparedStatement()方法，内部执行execute(sqlType, preparedStatementUnits, Collections.singletonList(parameters), executeCallback)。preparedStatementUnits含有数据源、SQL、preparestatement。将第一个任务从preparedStatementUnits分离出来，第一个任务同步执行，其他任务异步执行。由同步执行核心代码和异步执行核心代码可知，最终都是调用executeInternal()，跟读这个方法的源码可知：最终就是在目标数据库表上执行PreparedStatement的execute***()方法；且在执行前会利用google-guava的EventBus发布BEFORE_EXECUTE的事件（执行完成后，如果执行成功还会发布EXECUTE_SUCCESS事件，如果执行失败发布EXECUTE_FAILURE事件）。
2.1异步执行。ExecutorEngine的唯一成员executorService，是一个listen的线程池（guava中用于支持listenfuture的线程池）。使用该线程池异步批量执行executeInternal(sqlType, each, parameterSets, executeCallback, isExceptionThrown, dataMap);得到结果后封装。
2.2同步执行。直接调用executeInternal函数。
